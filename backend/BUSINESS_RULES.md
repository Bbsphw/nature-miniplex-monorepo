# กฎทางธุรกิจและเงื่อนไขของระบบ (Business Rules)

[⬅️ กลับหน้า Backend](./README.md) | [🏠 กลับสู่หน้าหลัก](../README.md)

ระบบ Nature MiniPlex ถูกออกแบบมาเพื่อตอบสนองต่อเงื่อนไขการทำงานจริงของโรงภาพยนตร์ เอกสารฉบับนี้รวบรวม **Business Rules (กฎทางธุรกิจ)** สำคัญที่นักพัฒนาทุกคนต้องคำนึงถึงเมื่อพัฒนาระบบ

---

## 1. การจัดการสิทธิ์ผู้ใช้งาน (Role-based Access Control)

ระบบ Backend มีการจำแนกสิทธิ์ผู้ใช้งานออกเป็น 2 ระดับ (Roles) ได้แก่ `Owner` และ `Staff`

- **Owner (เจ้าของ/ผู้จัดการ)**:
  - มีสิทธิ์เข้าถึง **ทุกฟีเจอร์** ในระบบ
  - สามารถสร้าง/แก้ไข/ลบ ข้อมูลระดับ Master Data ได้ (ภาพยนตร์, สาขาโรงภาพยนตร์, รอบฉาย, สร้างบัญชีพนักงาน)
  - มีสิทธิ์เรียกดูข้อมูลรายงานสรุปรายได้ (Reports)
- **Staff (พนักงานหน้าเคาน์เตอร์)**:
  - **ไม่มีสิทธิ์** เข้าถึงหรือแก้ไข Master Data และ Reports
  - หน้าที่หลักคือช่วยเหลือลูกค้าและจัดการการเริ่มต้น/ยกเลิกรายการจองหน้าเคาน์เตอร์ (ถ้ามี)
  - สามารถดูรอบฉายที่เปิดอยู่ได้

*การบังคับใช้กฎนี้*: ควบคุมด้วย Attribute `[Authorize(Roles = "Owner")]` ใน API Controllers

## 2. การจองตั๋ว (Ticket Booking)

### 2.1 กฎการจองที่นั่ง
- ลูกค้าไม่สามารถจองที่นั่งที่มีสถานะ "ถูกจองแล้ว (Booked/Sold)" ในรอบฉายเดียวกันได้
- ราคาของตั๋วในแต่ละที่นั่งจะถูกคำนวณจาก: `BasePrice` (ราคาภาพยนตร์) * `PriceMultiplier` (ตัวคูณราคาของรอบฉาย) * ตัวคูณประเภทที่นั่ง (เช่น ที่นั่ง VIP อาจจะมีราคาแพงกว่า)
- **การยกเลิก**: (ถ้ามีในอนาคต) การยกเลิกการจอง จะไม่ใช้การลบข้อมูล (Hard Delete) แต่จะใช้สถานะ (ItemStatus) ปรับเป็น `Canceled` เพื่อเก็บประวัติการทำรายการ

### 2.2 การป้องกันการจองซ้ำซ้อน (Concurrency Prevention)
เนื่องจากอาจมีลูกค้าหลายคนเห็นที่นั่งว่างพร้อมกันและกดจอง ระบบจึงใช้ **Filtered Unique Index** ร่วมกับ **Optimistic Concurrency** เพื่อป้องกัน Race Conditions:
- ในการสร้างรายการจองใหม่ (Insert Booking) Database จะมี Filtered Unique Index บน `(ShowtimeId, SeatId) WHERE ItemStatus = 'Active'` ในตาราง `BookingItem`
- หากลูกค้า 2 คนจองพร้อมกัน คนที่ 1 บันทึกสำเร็จ
- คนที่ 2 ที่พยายามจองที่นั่งเดียวกัน จะเจอข้อผิดพลาดระดับ Database (Unique Constraint Violation) ทันที 
- นอกจากนี้ยังใช้ `RowVersion` บน `BookingItem` ในกรณีที่มีการอัพเดท/ยกเลิกรายการจองในอนาคต เพื่อป้องกันการแก้ไขข้อมูลซ้อนทับกัน
- กฎนี้รับประกันว่า **ไม่มีทางที่จะเกิดปัญหาการจองที่นั่งซ้ำซ้อนกันอย่างเด็ดขาด 100%**

## 3. การจัดการข้อมูลระดับองค์กร (Audit & History)

- **Audit Trail (Action Logs)**: ทุกการกระทำที่สำคัญของพนักงาน (Owner/Staff) เช่น การล็อกรอบฉาย, การลบภาพยนตร์, การสร้างสาขา จะต้องถูกบันทึกลงตาราง `ActionLogs` เสมอ เพื่อการตรวจสอบย้อนหลัง
- **Soft Deletion**: ข้อมูลหลักอย่างภาพยนตร์ (`Movies`) และรอบฉาย (`Showtimes`) ห้ามใช้คำสั่ง `DELETE` ลบออกจากฐานข้อมูล แต่จะใช้แฟล็ก `IsActive = false` (Soft Delete) แทน เพื่อไม่ให้กระทบกับข้อมูลใบเสร็จหรือประวัติการจองตั๋วในอดีต
